Editorial

Presentation style guide for TXP Magazine article contributions

Use this guide when formatting content for magazine publication. We use Textile whenever possible (or at least convenient), so guides are written to demonstrate Textile use, along with the specific selectors and attributes, as needed. When Textile can’t be used (e.g. syntax doesn’t exist for a given style) we use straight HTML5, and also when it’s perhaps more intuitive to do so (complex tables, for example, can be confusing with Textile). Note: If you’re looking for guides about writing and editing copy, you want the Textpattern editorial style guide.

Context: TXP, authors, editors, Textile, formatting, markup, guide, elements, attributes

A-hem! This document is written with the assumption that you may not have used Textpattern before. Yes, we make allowances for that because not all magazine topics (e.g. reviews of other services/tools useful to freelancers) require Textpattern experience. So if you are experienced, thank you for tolerating the extra details given to make concepts like the admin-side layout and function more clear.

Overview of article submission process

Before you dive into this guide, make sure you even need to. The page you’re reading now is relevant at step 6 below, so if you’re not at step 6, stop and go back to the beginning:

  1. Draft a short summary of the idea you'd like to publish in TXP Magazine.
  2. Send it to magazine editors following the article contribution instructions (the only way we'll consider it).
  3. Wait patiently for a reply.
  4. If the idea is accepted, start a Google Document, using this template (you can only access it when you've been given rights to), and share it with your assigned editor.
  5. Outline your article, with summary statements at each point, and share with editor to ensure it's headed the right direction.
  6. Finish your draft, including indication of any code samples, visuals, or other presentational elements to be added. Begin (or finish) formatting your content with Textile using this guide.
  7. The article will get scheduled in the editorial calendar. (Date depends on editorial planning and slot availability.)
  8. You'll get an author account in the CMS to create and maintain your bio page. (Published at the same time as your article.)

Article structure and the Textpattern Write panel

Magazine website templates are created using HTML5. All presentation elements you will ever need are already defined and described in this guide. You only need to ensure that your article is formatted correctly by using the appropriate markup syntax and associated style attributes, as they are explained here. Everything corresponds to three editing fields in the Write panel of the CMS: Title, Excerpt, and Body.

Title

We use sentence-case on all titles (and body headers too). You just enter the title text, and that’s it. No Textile and regular markup is used.

Excerpt

Excerpts are the bold paragraphs that appear under the title (and author) of an article, like you see at top of this page. You may often need to italicise (__italicise__) or emphasise (_emphasise_) words, or add one or more links ("link":url). The following sentence, for example…

This represents a _sentence_ in an excerpt, having "this link":URL.

…would render as…

This represents a sentence in an excerpt, having this link.

That’s how Textile rolls. Simple.

Body

The body is where the rest of your content will go. We’ll cover the elements that are likely to be used most often, and maybe some special situations we’d want to employ for semantic reasons.

Block elements

Headers

Magazine articles use markup headers h1 to h4, and h1 is only used on titles automatically, so you will only ever use headers h2 through h4 in your body copy. We avoid using h5 and h6 as an editorial rule.

Paragraphs

A nice thing about Textpattern’s use of Textile is you don’t have to explicitly use Textile’s paragraph syntax (p.); you simply write your text, and the CMS will add p tags automatically.

There are two exceptions. First is when needing to use a selector on the paragraph (described next). The other time is when it follows a multi-paragraph blockquote (described later under Blockquotes).

Custom paragraphs (via presentational selectors)

If you’re not familiar with HTML markup, selectors are id and class attributes for HTML tags that can be used to tie presentational CSS styling to content with custom attribute values. For magazine articles, there are only a few situations requiring custom paragraph styles, and thus Textile paragraph notation at all.

Question–answer interviews and other author–subject conversations

The W3C specs for HTML5 make clear that blockquotes are not appropriate for marking up back-and-forth conversations, which question–answer interviews are when the conversation directly alternates between interviewer and interviewee. The W3C specs for conversations recommend using styled paragraphs instead:

This specification does not define a specific element for marking up conversations, meeting minutes, chat transcripts, dialogues in screenplays, instant message logs, and other situations where different players take turns in discourse.

Instead, authors are encouraged to mark up conversations using p elements and punctuation. Authors who need to mark the speaker for styling purposes are encouraged to use span or b. Paragraphs with their text wrapped in the i element can be used for marking up stage directions.

Since we often use question–answer and other conversational exchanges in the magazines Community Spotlight articles, we mark up the author’s dialogue in bold (**bold**) and the interviewee responses with p(answer)..

For example, this exchange marked-up with Textile…

**We hear you're a rolling stone that still collects moss. How is that possible?**

p(answer). I guess because I prefer to roll slowly in moist and shady ground.

Renders as this…

<p><b>We hear you’re a rolling stone that still collects moss. How is that possible?</b></p>
<p class="answer">I guess because I prefer to roll slowly in moist and shady ground.</p>

Note that making **bold** text with Textile requires surrounding the copy with double asterisks, which is semantically different than using single asterisks to make *strong* text. Even though they produce the same visual result when rendered, they result in different HTML elements—b and strong, respectively—and thus communicate different meanings to (and thus responses from) text-reading technology that website visitors may rely on. In other words, this is an important accessibility factor, one authors need to be mindful of at all times. Ensure you use double asterisks when you just need bold text, like in this case, because we don’t want to emphasise author questions over subject responses.

Blockquotes

Blockquotes are paragraphs quoted from an external source. Blockquotes should ideally be cited with a URL to the source, but if no URL exists, blockquote markup is still valid if a reference to an original print publication of the source can be provided in it’s place.

Textile can be used for blockquotes easily, and there’s a convention for both single-paragraph blockquotes (bq.) and multi-paragraph blockquotes (bq..).

Single-paragraph blockquotes

A single-paragraph blockquote (bq.) used between two normal paragraphs would be marked up with Textile like this…

A regular paragraph.

bq. Start of the blockquote paragraph...

Another regular paragraph.

According to the HTML5 standards, you don't have to use p tags inside a blockquote, but Textpattern adds them anyway, so single-paragraphs will result in HTML markup like this...

<blockquote>
    <p>Start of the blockquote paragraph...</p>
</blockquote>

And render as formatted content like…

Start of the blockquote paragraph...

Blockquotes are visually styled exactly like the custom paragraphs (p(answer).) described earlier. We want the uniformity in presentation while abiding by the W3C’s recommendations for distinguishing between blockquotes and regular non-cited conversations. The difference is under the surface—machine-readable semantics—and important for text-reading technology.

Multi-paragraph blockquotes

For multi-paragraph blockquotes (bq..), you do need to use the p. on the paragraph immediately following the blockquote to escape the paragraph from the blockquote code. (This is the second situation when the Textile syntax for paragraphs must be used in Textpattern, as mentioned at the start of the Paragraphs section.)

Here’s a typical structure for a multi-paragraph blockquote using Textile…

A regular paragraph...

bq.. First blockquote paragraph...

Second blockquote paragraph...

Et cetera

p. First regular paragraph following multi-paragraph blockquote.

The above will result in HTML markup like this...

<blockquote>
    <p>First blockquote paragraph...</p>
    <p>Second blockquote paragraph...</p>
    <p>Et cetera</p>
</blockquote>

Cited blockquotes

Citing the source from where the quote is being taken from should always be done if the source has a URI than can be used. This requires both the cite element and the cite="" attribute, used in relation to the blockquote element.

The Textile convention for the cite element is ??cite??, where "cite" is the named source of the quote. The Textile convention for the cite="" attribute in relation to the blockquote is bq.:url, where "url" is the actual URL to the source. Using it all together, your Textile will look like this…

Leading paragraph with ??cited reference??:

bq.:url Blockquote paragraph...

And this is the resulting markup...

<p>Leading paragraph with <cite>cited reference</cite>:</p>
<blockquote cite="url">
    <p>Blockquote paragraph…</p>
</blockquote>

See the earlier section, Question–answer interviews, which uses a cited blockquote to reference the W3C specs for HTML5, and demonstrates the rendered presentation.

The same is true for multi-paragraph blockquotes, just making sure to use bq..:

Leading paragraph with ??cited reference??:

bq..:url First blockquote paragraph.

Second blockquote paragraph.

Asides

The aside element is what we use for pull quotes and Editor notes.

Pull quote asides

Queequeg, don’t eat me!

These can be used in articles as a way to make a series of paragraphs a little more interesting. Styles exist for left and right positions with paragraphs. The one you see here is right-aligned, and is done using <aside class="aright">.

A pull quote can be positioned to the left too using <aside class="aleft">.

Editor note asides

Editor notes are side notes that might be added by editors post publication if new (helpful and relevant) information comes to attention and seems worth adding. Editor notes are full-width and created using…

<aside class="editor">
    <b>Editor's note:</b> Some of the above is true, and some isn't.
</aside>

which renders…

Editor’s note: Some of the above is true, and some isn’t.

Inline elements

Inline elements (or text level elements), are essentially those used within block element copy. Out of all that exist, the ones you will probably use the most are:

  • italic __italic__)
  • emphasis (_emphasis_)
  • bold (**bold**)
  • strong (*strong*), meaning important
  • links ("links":url)

See the quick reference table in the sidecolumn of this page for Textile equivalents to many inline elements you might use.

Notes on links

Links are pretty straightforward, but there’s a couple of things to be aware.

Links with title attributes

If you want title hovers for our links, you can still do this with Textile by simply adding the title text in parenthesis after the link label: "Textpattern(The Textpattern website)":http://textpattern.com.

Absolute and relative links

Use a relative (local) URL if you’re linking to another page within TXP (e.g. "Write for us":contribute/write-for-us), which always begins with the section title.

Use an absolute URL when linking outside of TXP (e.g. "Textpattern":http://textpattern.com).

Escaping links

There are certain situations where links need to be escaped from associated punctuation for formatting to work properly.

Escaping em dashes (before and after): TXP uses em dashes in sentence structure with no surrounding spaces. If your copy requires a link immediately next to an em dash, en dash, etc., Textile syntax will simply display as non-rendered text. This is easily corrected by surrounding the Textile link syntax with straight brackets. For example, this…

...tail end of a sentence—["this link":URL].

will render correctly as…

…tail end of a sentence—this link.

Without the straight brackets, the Textile syntax and associated URL would just appear as non-rendered text. You can markup the link using regular HTML, if you prefer, which also works as expected.

Escaping quotation marks: If you ever need (or more likely want) to put a link on a quote, again use the straight brackets between the quotes marks of the quoted text and those of the Textile link syntax. So this…

She said, "["Hey, mofo!":url]".

turns out okay, like this…

She said, "Hey, mofo!".

Image

Images will be used in two contexts in the magazine: alone (with or without link) or as figures (links, if used, in the figure’s caption only). Each context will share the same four predetermined sizes (widths as percentages) handled in the CSS: 25%, 35%, 55% and 100%. The 25, 35, and 55% widths will always be floated left or right. All images must be pre-processed accordingly before loaded into the CMS; meaning they should be cropped to a pixel width slightly greater than the targeted percentage width (Table 2).
